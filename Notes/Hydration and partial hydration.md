# Hydration and partial hydration

## Overview

**Hydration** is the process by which a server-rendered HTML page becomes **fully interactive** in the browser. It is the critical bridge between SSR (which delivers fast, visible HTML) and the interactive React app the user expects to use.

Understanding hydration — and its cost — is essential for building performant React/Next.js applications.

---

## What Is Hydration?

When SSR delivers HTML, the page is **visually complete but not interactive**. The HTML has buttons, forms, and links, but clicking them does nothing. JavaScript hasn't run yet.

Hydration is React "waking up" the static HTML by:
1. **Downloading** the component JavaScript bundle.
2. **Re-rendering** the component tree in memory (matching the server output).
3. **Attaching event listeners** to the existing DOM — no DOM replacement.
4. **Making the page interactive**.

```
SSR Phase:        Server → [Full HTML] → Browser paints page (Fast, but static)
Hydration Phase:  Browser downloads JS → React runs → Event listeners attached (Page is now interactive)
```

---

## The Hydration Timeline

```
─────────────────────────────────────────────────────
  0ms    Browser receives HTML from server
         ← User SEES the page (FCP / LCP met)

  ~0ms   Browser starts downloading JS bundles (parallel)

 300ms   JS bundle downloaded and parsed

 400ms   React hydration begins
         → React renders component tree to match DOM
         → Attaches onClick, onChange, etc.

 500ms   Hydration complete
         ← Page is NOW interactive (TTI met)

During 0-500ms: User SEES the page but cannot interact.
                Clicking buttons does nothing.
                This "uncanny valley" must be minimized.
─────────────────────────────────────────────────────
```

The period between FCP (page visible) and TTI (page interactive) is the **hydration gap**. This is where users click a button and nothing happens — a very frustrating UX.

## Hydration in Next.js

### React Server Components: Zero Hydration Cost

In Next.js App Router, **Server Components do not need hydration at all**. Their code never ships to the browser. Only **Client Components** (`'use client'`) require hydration:

```typescript
// app/dashboard/page.tsx — Server Component
// ✅ No hydration needed — this code runs ONLY on the server
export default async function DashboardPage() {
  const data = await fetchData()

  return (
    <main>
      <h1>Dashboard</h1>     {/* Static — no hydration cost */}
      <DataTable data={data} /> {/* Server Component — no hydration */}
      <RefreshButton />      {/* Client Component — WILL hydrate */}
    </main>
  )
}

// app/dashboard/RefreshButton.tsx — Client Component
'use client'
// ✅ Only THIS component needs hydration — much smaller scope
export default function RefreshButton() {
  return <button onClick={() => window.location.reload()}>Refresh</button>
}
```

This is the key performance advantage of React Server Components — you ship and hydrate only the interactive parts.

### Hydration Mismatch Error

A hydration mismatch occurs when the HTML generated by the server doesn't match what React renders in the browser. React throws an error and falls back to a full client-side re-render.

```typescript
// ❌ This WILL cause a hydration mismatch
// Server renders: "Hello" (no window object on server)
// Browser renders: "Hello" + random number (Math.random() differs)
export default function ProblematicComponent() {
  const id = Math.random() // Different each render → MISMATCH
  return <div>ID: {id}</div>
}

// ✅ Fix: Use useId() for stable IDs, or move dynamic parts to useEffect
'use client'
import { useId, useState, useEffect } from 'react'

export default function FixedComponent() {
  const id = useId() // Stable, consistent between server and client
  const [clientOnly, setClientOnly] = useState<string | null>(null)

  useEffect(() => {
    // This only runs in the browser — safe for browser-only values
    setClientOnly(window.navigator.userAgent.slice(0, 10))
  }, [])

  return (
    <div>
      <span>ID: {id}</span>
      {clientOnly && <span>Browser: {clientOnly}</span>}
    </div>
  )
}
```

## Partial Hydration

**Partial hydration** is the strategy of only hydrating the **interactive parts** of a page, leaving the static parts as plain HTML without attaching any React JavaScript to them.

This is exactly what Next.js App Router achieves through Server vs Client Component separation.

### Old approach (full hydration)

```
Traditional React SSR:
  Server renders entire page → HTML
  Client downloads ENTIRE React app bundle
  Client hydrates EVERY component (even static ones)
  → Wasted hydration on components that never change
```

### New approach (partial hydration via RSC)

```
Next.js App Router:
  Server renders Server Components → static HTML (most of the page)
  Client downloads ONLY Client Component bundles
  Client hydrates ONLY Client Components
  → Much smaller JS payload, faster interactivity
```

```typescript
// Practical example of the difference:

// FULL HYDRATION (old Pages Router):
// The entire page below hydrates — including the static header and footer

// PARTIAL HYDRATION (App Router):
// app/product/page.tsx
export default async function ProductPage({ params }) {
  const product = await getProduct(params.id) // Server-side only

  return (
    <div>
      {/* Server Components — zero JS cost on client */}
      <StaticHeader />
      <ProductDescription product={product} />
      <ProductImages images={product.images} />

      {/* Client Components — only these hydrate */}
      <AddToCartButton productId={product.id} />  {/* 'use client' */}
      <QuantitySelector />                          {/* 'use client' */}
      <WishlistToggle productId={product.id} />   {/* 'use client' */}
    </div>
  )
}
```

On this page, only 3 small Client Components hydrate. The rest of the page (description, images, header) has zero JS hydration cost.

### Deferring hydration with dynamic imports

You can lazy-hydrate heavy Client Components — defer them until they're needed:

```typescript
import dynamic from 'next/dynamic'

// This component's JS is not downloaded until it's needed
const HeavyModal = dynamic(() => import('@/components/HeavyModal'), {
  ssr: false,  // Don't SSR this — purely client-side
  loading: () => <p>Loading...</p>,
})

export default function Page() {
  const [showModal, setShowModal] = useState(false)

  return (
    <>
      <button onClick={() => setShowModal(true)}>Open Modal</button>
      {showModal && <HeavyModal />}  {/* Only hydrates when modal is opened */}
    </>
  )
}
```

## Pros and Cons

| Aspect | Full Hydration | Partial Hydration (RSC) |
| :--- | :--- | :--- |
| **JS bundle size** | Large (entire app) | Small (only interactive parts) |
| **Time to Interactive** | Longer | Shorter |
| **Hydration mismatch risk** | Higher | Lower (fewer hydrated components) |
| **Complexity** | Lower | Medium |
